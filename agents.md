# Architectural Overview

G33kSeek is built around a **query provider system**.

User input is interpreted and routed to a provider responsible for generating results.

Example:

```
QueryEngine
   ├── AppSearchProvider
   ├── FileSearchProvider
   ├── CalculatorProvider
   ├── WebSearchProvider
   ├── ContentSearchProvider
   ├── AiPromptProvider
   └── CommandProvider
```

Providers should:

- Accept a query string
- Return ranked results
- Execute an action when selected

Providers should remain **loosely coupled** so new functionality can be added easily.

Agents should prefer placing new code in the following folders so the repository remains organised and predictable.

```
/Providers
```

Guidelines:

- **Providers** implement query functionality (calculator, file search, AI prompts, etc).
- **Services** contain reusable logic (indexing, search orchestration, AI communication, etc).
- **Views** contain Avalonia `.axaml` UI definitions.
- **ViewModels** implement the UI logic using MVVM.
- **Commands** contain executable actions triggered from results.
- **Models** contain simple data structures used by providers and services.

Avoid scattering classes across the repository.  
If unsure where something belongs, prefer **Services** or **Providers**.

# Query Prefix System

The first character(s) of the query determines the provider.

| Prefix | Mode | Example |
|------|------|------|
| *(none)* | File/App search | `rider` |
| `=` | Calculator | `=2+2` |
| `?` | Web search | `?avalonia docs` |
| `??` | File content search | `??TODO` |
| `@` | AI prompt | `@summarise this text` |
| `>` | Commands | `>shutdown` |

When no prefix is used, the launcher performs **file and application search**.

# Global Hotkey Implementation

Global keyboard shortcuts should be implemented using the **SharpHook** library.

SharpHook provides reliable cross‑platform global keyboard hooks and works well with .NET and Avalonia.

Agents should prefer **SharpHook** rather than implementing platform‑specific keyboard hooks.

Example responsibilities:

- Register a global launcher shortcut (e.g. `Ctrl+Space` or user configurable).
- Trigger the launcher window when the shortcut is pressed.
- Ensure the hook runs off the UI thread where appropriate.

Keep the implementation lightweight and avoid blocking the hook thread.

# DTC.Core Usage

This project includes **DTC.Core** as a submodule and agents should prefer using its helpers rather than re‑implementing functionality.

Agents should also consider whether new generic helpers belong in **DTC.Core** instead of G33kSeek. If code is broadly reusable and not specific to launcher behaviour or Avalonia UI, it is a good candidate for **DTC.Core**.

File-system objects should usually be represented as **FileInfo** and **DirectoryInfo** rather than raw strings.

Agents should prefer:

```
FileInfo
DirectoryInfo
DTC.Core extension methods
TempFile
```

over passing file-system paths around as plain strings where a richer type is sensible.

Useful helpers include:

File system helpers:

```
string.ToDir()
string.ToFile()
dir.GetDir("a/b")
dir.GetFile("x.txt")
```

Safe enumeration:

```
DirectoryInfo.TryGetDirs(...)
DirectoryInfo.TryGetFiles(...)
DirectoryInfo.TryGetContent(...)
```

Deletion:

```
DirectoryInfo.TryDelete()
FileInfo.TryDelete()
FileSystemInfo.TryDelete()
```

Opening items:

```
FileInfo.OpenWithDefaultViewer()
Uri.Open()
```

Existence checks:

```
fileInfo.Exists()
directoryInfo.Exists()
```

# ReviewG33k-Friendly Defaults

These are lightweight habits that help keep ReviewG33k findings low without over-optimising for any single tool.

- Keep files focused and avoid duplicate helper code where an existing service or extension method already fits.
- Prefer simple MVVM boundaries: keep view logic in views, state/commands in view-models, and reusable logic in services/providers.
- Avoid unused `using` directives, dead private members, and one-off fields that can just be locals.
- Prefer `readonly` for fields that are only assigned during construction.
- Keep async code honest:
  - prefer `Task` / `Task<T>` over `async void`
  - use the `Async` suffix for async methods unless a framework/interface contract prevents it
  - avoid wrapping async work in `Task.Run(...)` unless there is a clear need
- When writing numeric values to files or persisted text, prefer invariant culture formatting.
- Add class-level XML docs to reusable public types:
  - `<summary>` = what it does
  - `<remarks>` = why it is useful here
- Keep Avalonia bindings tidy:
  - prefer typed bindings/context hints where practical
  - avoid unnecessary wrapper panels/containers
- Preserve clean formatting:
  - blank lines between methods
  - no stray blank lines between consecutive brace-only lines
- If behaviour changes, update tests. If the user-facing workflow changes noticeably, also update the README.
